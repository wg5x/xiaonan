# AI 标书写作平台 MVP PRD

## 1. 项目目标

本项目第一版目标是做出一条可上线、可演示、可销售验证的 AI 标书写作主链路：

> 用户上传招标文件，系统解析关键要求，生成标书目录和章节正文，用户编辑确认后导出 Word，并完成基础废标风险检查。

MVP 不追求一次性覆盖完整 SaaS 平台，而是优先验证三个核心问题：

1. AI 是否能准确解析招标文件中的评分标准、资质要求、废标条款和技术参数。
2. AI 是否能基于解析结果生成可用的标书目录和章节正文。
3. 系统是否能稳定导出格式可用的 Word 标书。

## 2. 目标用户

### 2.1 主要用户

- 中小企业投标负责人。
- 标书制作人员。
- 代写标书服务团队。

### 2.2 典型场景

用户拿到一份招标文件后，需要快速完成以下工作：

- 看懂招标要求。
- 提取评分点和废标风险。
- 生成标书目录。
- 编写技术、商务、服务等章节。
- 导出 Word 后继续人工完善。

## 3. MVP 范围

MVP 只包含主链路必需能力。

| 模块 | 是否进入 MVP | 说明 |
| --- | --- | --- |
| 用户登录 | 是 | 可先支持账号密码登录，不做复杂第三方登录 |
| 项目管理 | 是 | 一个项目对应一份标书 |
| 招标文件上传 | 是 | 优先支持 PDF |
| PDF 文本解析 | 是 | 第一版优先保证文字型 PDF，扫描件 OCR 可作为增强项 |
| 招标文件 AI 解读 | 是 | 提取评分标准、资质要求、废标条款、技术参数等 |
| 智能目录生成 | 是 | 基于解析结果生成目录，支持人工调整 |
| 分章节 AI 写作 | 是 | 支持按章节生成、重新生成、人工编辑 |
| 基础在线编辑 | 是 | 支持正文编辑、标题层级、简单表格和图片占位 |
| Word 导出 | 是 | 固定 1-2 套标准模板，保证可交付 |
| 废标风险检查 | 是 | 对硬性要求做基础核对，输出风险清单 |
| Token 统计 | 是 | 记录调用量和消耗明细，先不接真实支付 |
| 管理后台 | 部分 | 只做最小后台：用户、项目、Token 记录、系统配置 |

## 4. 不做范围

以下功能不进入 MVP，避免第一版过重：

- 标讯搜索、商机推送、招标信息订阅。
- PPT 自动生成。
- AI 生图和批量配图。
- 多人协作、评论、批注、复杂权限。
- 完整企业知识库和向量检索。
- 历史标书高级复用。
- 支付宝、微信支付、优惠码、自动开票。
- 大量行业模板库。
- 格式模板自定义。
- PDF 直接导出。
- 自研大模型。

这些能力可以进入二期或三期规划，但不能阻塞 MVP 上线。

## 5. 核心用户流程

1. 用户登录系统。
2. 创建标书项目。
3. 上传招标文件 PDF。
4. 系统解析 PDF 文本。
5. AI 提取招标关键信息：
   - 项目名称。
   - 投标截止时间。
   - 评分标准。
   - 资质要求。
   - 废标条款。
   - 技术参数。
   - 商务条款。
6. 用户查看并确认解析结果。
7. 系统生成标书目录。
8. 用户调整目录。
9. 用户选择章节并触发 AI 写作。
10. 用户在编辑器中修改内容。
11. 系统执行废标风险检查。
12. 用户处理风险提示。
13. 导出 Word 标书。

## 6. 功能需求

### 6.1 用户登录

第一版只需要满足基本访问控制。

| 功能 | 要求 |
| --- | --- |
| 注册 | 支持手机号或邮箱注册，二选一即可 |
| 登录 | 账号密码登录 |
| 密码存储 | 使用 bcrypt 加密 |
| 会话 | 使用 JWT 或服务端 Session |
| 权限 | 普通用户只能访问自己的项目 |

### 6.2 项目管理

| 功能 | 要求 |
| --- | --- |
| 创建项目 | 填写项目名称，可选填写客户/行业 |
| 项目列表 | 展示项目名称、创建时间、更新时间、状态 |
| 项目详情 | 展示招标文件、解析结果、目录、章节内容 |
| 删除项目 | 软删除，避免误删数据无法恢复 |

项目状态建议：

- 未上传招标文件。
- 解析中。
- 待确认解析结果。
- 写作中。
- 待检查。
- 已导出。

### 6.3 招标文件上传与解析

| 功能 | 要求 |
| --- | --- |
| 文件类型 | MVP 优先支持 PDF |
| 文件大小 | 建议限制 50MB 以内 |
| 解析方式 | 文字型 PDF 使用 PyMuPDF |
| OCR | 扫描件 OCR 可放入增强项，MVP 可提示“暂不支持扫描件” |
| 解析结果 | 保存原始文本，便于后续 AI 调用和人工排查 |

验收标准：

- 可上传普通文字型 PDF。
- 系统能提取主要文本内容。
- 解析失败时给出明确错误提示。
- 原文件和解析文本能关联到项目。

### 6.4 招标文件 AI 解读

AI 解读结果必须结构化保存，不能只返回一段普通总结。

建议字段：

| 字段 | 说明 |
| --- | --- |
| 项目名称 | 从招标文件中提取 |
| 招标单位 | 从招标文件中提取 |
| 投标截止时间 | 如能识别则提取 |
| 评分标准 | 按评分项拆分，包含分值和说明 |
| 资质要求 | 企业资质、人员资质、业绩要求等 |
| 废标条款 | 明确标记硬性风险 |
| 技术参数 | 关键技术指标和响应要求 |
| 商务条款 | 报价、付款、服务、交付等要求 |
| 需人工确认项 | AI 不确定的内容必须标出 |

验收标准：

- 解读结果按模块展示。
- 每个关键要求尽量关联原文片段。
- 用户可以人工修改或标记确认。
- AI 不确定内容不能伪装成确定结论。

### 6.5 智能目录生成

系统根据评分标准和招标要求生成标书目录。

| 功能 | 要求 |
| --- | --- |
| 目录生成 | 生成一份默认目录 |
| 目录调整 | 支持新增、删除、重命名、排序章节 |
| 章节层级 | MVP 支持二级目录即可 |
| 目录来源 | 目录应优先覆盖评分点和硬性要求 |

验收标准：

- 目录能覆盖主要评分点。
- 用户能调整目录。
- 每个章节能进入后续 AI 写作流程。

### 6.6 分章节 AI 写作

分章节写作是 MVP 的核心。

| 功能 | 要求 |
| --- | --- |
| 单章生成 | 用户选择一个章节后生成正文 |
| 重新生成 | 支持重新生成当前章节 |
| 上下文 | 生成时带入招标要求、评分点、目录、项目基础信息 |
| 内容保存 | 每次生成结果保存到项目章节 |
| 人工编辑 | 用户可直接修改 AI 生成内容 |

AI 写作要求：

- 语气专业，适合标书。
- 避免明显虚假承诺。
- 对缺失的企业资料使用占位提示，不编造资质。
- 与评分标准和废标条款保持对应。

验收标准：

- 至少能生成 5 个不同章节的正文。
- 生成内容能保存、修改、再次打开。
- 缺少必要资料时，系统提示用户补充，而不是编造。

### 6.7 基础在线编辑

MVP 编辑器只做必要能力，不做复杂 Word 编辑器。

| 功能 | 要求 |
| --- | --- |
| 正文编辑 | 支持段落文本编辑 |
| 标题层级 | 支持一级、二级、三级标题 |
| 基础格式 | 加粗、列表、表格 |
| 图片 | 可先支持图片占位或简单上传 |
| 自动保存 | 建议支持定时保存或离开保存 |

不要求：

- 完整 Word 级排版能力。
- 复杂页眉页脚编辑。
- 拖拽式图文混排。

### 6.8 Word 导出

Word 导出是 MVP 的交付关键。

| 功能 | 要求 |
| --- | --- |
| 模板数量 | MVP 固定 1-2 套标准模板 |
| 字体 | 正文宋体四号或可配置为固定样式 |
| 标题 | 自动编号，层级清晰 |
| 页边距 | 使用标准页边距 |
| 页码 | 自动页码 |
| 目录 | 可先导出静态目录，后续支持自动目录 |
| 文件命名 | 项目名称 + 导出时间 |

验收标准：

- 可生成可打开的 `.docx` 文件。
- 章节顺序与用户确认的目录一致。
- 正文、标题、表格基本样式稳定。
- 不出现乱码、内容丢失、明显格式错乱。

### 6.9 废标风险检查

废标检查是 MVP 的差异化重点之一。

| 功能 | 要求 |
| --- | --- |
| 检查对象 | 招标文件中的硬性要求 |
| 检查方式 | 将硬性要求与当前标书内容进行匹配 |
| 风险等级 | 高、中、低 |
| 输出形式 | 风险清单 |
| 人工处理 | 用户可标记“已处理”或“忽略” |

风险类型示例：

- 缺少资质响应。
- 缺少签字盖章提示。
- 未响应关键技术参数。
- 未覆盖评分项。
- 投标截止时间、保证金等关键信息未提醒。

验收标准：

- 系统能输出结构化风险清单。
- 每条风险包含原因和建议处理方式。
- 用户可以回到对应章节修改。

### 6.10 Token 统计与用量控制

MVP 先记录用量，不接真实支付。

| 功能 | 要求 |
| --- | --- |
| 调用记录 | 记录用户、项目、功能、模型、时间 |
| Token 记录 | 记录输入 Token、输出 Token、总 Token |
| 余额控制 | 可先做后台配置额度 |
| 超额处理 | 超出额度后禁止继续 AI 调用 |

注意：API Key 不得硬编码到代码仓库，应通过环境变量或密钥配置注入。

文件路径必须和用户、项目绑定，避免跨用户访问。

## 9. AI 能力设计

AI 能力不要散落在业务代码里，建议抽象为统一服务。

### 9.1 AI 调用类型

| 类型 | 输入 | 输出 |
| --- | --- | --- |
| 招标解读 | PDF 解析文本 | 结构化招标要求 |
| 目录生成 | 招标要求、评分标准 | 目录树 |
| 章节写作 | 章节名称、招标要求、上下文 | 章节正文 |
| 风险检查 | 招标硬性要求、标书内容 | 风险清单 |

### 9.2 输出要求

- 优先使用 JSON 结构化输出。
- 每次 AI 输出需要做格式校验。
- 校验失败时允许重试一次。
- 保存原始 AI 响应，便于排查，但敏感内容要限制访问。

### 9.3 安全要求

- API Key 只存在服务端。
- 前端不出现模型 Key 或中转 Key。
- 每次调用先检查用户额度。
- 设置单次最大 Token 上限。
- 记录异常消耗并提醒管理员。

## 10. 技术建议

MVP 可采用以下技术栈：

| 层级 | 建议 |
| --- | --- |
| 前端 | Vue 3 + Vite + Element Plus |
| 编辑器 | TinyMCE 或 Quill，选择一个即可 |
| 后端 | Python FastAPI |
| 数据库 | PostgreSQL |
| 缓存/队列 | Redis |
| 异步任务 | Celery 或 RQ |
| PDF 解析 | PyMuPDF |
| OCR | 二期增强，或作为可选能力接入 PaddleOCR |
| Word 生成 | python-docx |
| 文件存储 | 本地磁盘起步，后续迁移 MinIO/OSS |
| 部署 | Docker Compose + Nginx |

耗时任务建议异步执行：

- PDF 解析。
- AI 解读。
- 章节批量生成。
- Word 导出。

## 11. 计费与商业化边界

MVP 不接真实支付，但要为商业化留接口。

### 11.1 MVP 阶段

- 后台给用户配置额度。
- 记录 AI 调用 Token。
- 超额后限制 AI 调用。
- 可人工调整额度。

### 11.2 后续商业化

- 月度会员。
- 年度会员。
- Token 包充值。
- 企业版团队套餐。
- 微信支付、支付宝、对公转账。

## 12. 验收标准

MVP 完成的最低验收标准如下：

1. 用户可以登录并创建项目。
2. 用户可以上传一份文字型 PDF 招标文件。
3. 系统可以提取 PDF 文本。
4. 系统可以输出结构化招标解读结果。
5. 用户可以确认或修改解读结果。
6. 系统可以生成标书目录。
7. 用户可以调整目录。
8. 用户可以按章节生成标书正文。
9. 用户可以编辑并保存章节正文。
10. 系统可以基于废标条款输出风险清单。
11. 用户可以导出 Word 文件。
12. Word 文件能正常打开，内容完整，基础格式稳定。
13. 后台可以查看用户、项目和 AI 调用记录。
14. API Key 不出现在前端和代码仓库。

## 13. 二期规划

二期目标是提升复用能力和商业化能力。

- 企业资料库。
- 历史标书库。
- 模板库和占位符替换。
- 一键排版增强。
- 支付接入。
- 用户套餐和 Token 充值。
- 管理后台完善。
- OCR 扫描件支持。
- 评分自检增强。

## 14. 三期规划

三期目标是扩大平台能力。

- 多人协作。
- 项目权限管理。
- 多版本对比。
- PPT 自动生成。
- 图表生成。
- AI 生图。
- 批量配图。
- 格式模板自定义。
- 标书对比。

## 15. 主要风险

| 风险 | 影响 | 建议 |
| --- | --- | --- |
| PDF 解析质量不稳定 | 影响后续 AI 解读 | MVP 优先支持文字型 PDF，扫描件作为增强 |
| AI 编造企业资质 | 可能造成投标风险 | 缺资料时必须输出占位提示，不允许编造 |
| Word 导出格式不可控 | 影响交付体验 | 固定模板起步，不做复杂自定义 |
| 功能范围膨胀 | 导致延期 | 严格执行 MVP 不做范围 |
| Token 成本不可控 | 影响毛利 | 设置额度、单次上限和调用日志 |
| 用户资料敏感 | 存在合规风险 | 做权限隔离、日志脱敏、文件访问控制 |

## 16. 待确认事项

以下事项需要在开发前确认：

1. MVP 是否只支持文字型 PDF，扫描件 OCR 是否放到二期。
2. 第一版是否需要真实注册，还是内部账号即可。
3. 是否需要接入真实 ipsunion API，还是先用测试 Key。
4. Word 模板采用几套，是否已有标准模板样例。
5. 是否需要导入企业基础资料，如公司名称、资质、业绩案例。
6. 第一批试用客户属于哪个行业，以便优化提示词和模板。
7. Token 单价和试用额度如何设置。
